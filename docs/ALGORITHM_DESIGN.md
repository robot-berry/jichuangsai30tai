# 30TAI 目标瞄准与定距跟随算法设计说明

## 1. 项目定位

本项目面向 30TAI 开发板上的视觉目标跟随任务，目标不是重新复制完整的 YOLOv5 + HDMI + CAN 示例工程，而是抽取并整理一个可以集成到现有工程中的独立算法模块。

现有 PLin + SingleNet + HDMI 工程已经完成摄像头输入、YOLOv5 目标检测、HDMI 显示和 CAN 通信。本项目在此基础上补充两个核心能力：

1. 根据目标检测框中心控制云台，使目标尽量保持在图像中心。
2. 根据单目距离估计结果控制小车底盘，使小车与目标保持设定距离。

因此，本项目的输入不是原始图像，而是检测后得到的目标框、类别、置信度、距离估计值和图像尺寸；输出是云台位置增量和底盘电机速度命令。

## 2. 场景与约束

### 2.1 应用场景

本模块适用于固定类别目标的视觉跟随场景，例如自行车图片、目标板、装甲板或其他已知实际宽度的目标。系统通过摄像头实时获取图像，YOLOv5 检测目标位置，随后算法模块根据目标位置和距离生成控制命令。

典型运行链路如下：

```text
摄像头输入 -> YOLOv5 检测 -> 目标框筛选 -> 单目距离估计
          -> 云台瞄准控制 -> 小车定距跟随控制 -> CAN 输出
```

### 2.2 30TAI 上板约束

30TAI 上板运行时需要考虑以下限制：

- 算力资源有限，不适合额外加入复杂 ReID、深度估计或多目标跟踪网络。
- 原工程已经包含 YOLOv5 推理，新增模块应尽量轻量，避免明显增加推理延迟。
- 底盘控制通过 CAN 发送整数速度值，不能直接输出连续浮点控制量。
- 云台控制主要通过位置命令实现，适合使用图像误差到位置增量的方式，而不是完整连续伺服控制。
- 目标丢失时必须有安全策略，避免继续输出上一次的运动命令。

基于这些约束，本项目选择轻量算法组合，而不是引入复杂端到端控制模型。

## 3. 需求分析

### 3.1 功能需求

本模块需要完成以下功能：

- 从检测框中选择当前最合适的跟随目标。
- 根据目标中心与图像中心的偏差计算云台 yaw、pitch 调整量。
- 根据目标框宽度和相机标定参数估计目标距离。
- 根据当前距离与设定距离的误差计算底盘前进或后退速度。
- 目标短暂漏检时保持目标连续性，避免频繁切换目标。
- 目标连续丢失后停止底盘运动，防止小车误动。
- 输出日志，便于实车调试和参数整定。

### 3.2 性能需求

算法应满足以下工程指标：

- 单帧控制计算开销远小于 YOLOv5 推理开销。
- 控制输出为整数，能直接适配 CAN 发送格式。
- 参数集中在配置结构体中，便于实车调试。
- 与原工程解耦，便于单独编译、单元测试和后续迁移。
- 不改变原工程的检测模型、摄像头链路、HDMI 显示链路和 CAN 基础收发逻辑。

### 3.3 安全需求

实车运行时需要避免以下风险：

- 检测框跳变导致云台突然大幅摆动。
- 多目标场景下误切换目标导致小车方向突变。
- 目标丢失后底盘仍按旧命令前进。
- 距离估计异常导致速度过大。

因此，本项目在控制中加入死区、限幅、目标连续选择和目标丢失保护。

## 4. 智能算法选择

### 4.1 YOLOv5 目标检测

原工程已经使用 YOLOv5 完成目标检测，因此本项目不重复实现检测网络，而是复用检测输出。YOLO 系列算法适合嵌入式实时目标检测场景，能够在速度和精度之间取得较好平衡。

在本项目中，YOLOv5 的职责是输出候选框，算法模块只关心以下信息：

- 目标框左上角坐标和宽高。
- 目标类别。
- 检测置信度。
- 图像宽高。

这种设计可以保证算法模块与检测模型弱耦合，后续如果将 YOLOv5 替换为 YOLOv8、YOLOv10 或其他检测器，只要输出检测框格式一致，控制模块仍可复用。

### 4.2 轻量目标连续选择

原工程容易出现“每帧选择最大框”的问题：当画面中有多个目标，或者检测框发生抖动时，控制目标可能频繁切换。为降低切换，本项目参考多目标跟踪中的关联思想，但没有引入 DeepSORT 或 ByteTrack 的完整流程。

本项目采用轻量连续选择策略：

- 若上一帧已有目标，则优先选择与上一帧中心距离最近的框。
- 当新目标面积明显更大时，允许切换到更合理的目标。
- 目标短暂漏检时保留丢失计数，不立即清空目标状态。
- 连续丢失超过阈值后判定目标丢失。

这样做的原因是：30TAI 上已经运行检测模型，再引入完整跟踪网络会增加部署复杂度；而当前任务主要是单目标跟随，轻量关联策略已经能解决多数目标跳变问题。

### 4.3 单目距离估计

距离估计采用针孔相机模型。若目标实际宽度已知，检测框像素宽度为 `pixel_width`，相机等效焦距为 `focal_length_px`，则距离可按下式估计：

```text
distance_m = real_width_m * focal_length_px / pixel_width
```

其中：

- `real_width_m`：目标实际宽度，实测自行车图片宽度为 0.24 m。
- `focal_length_px`：通过实测标定得到的像素焦距。
- `pixel_width`：检测框在图像中的像素宽度。

该算法不需要额外深度相机，计算量极低，非常适合 30TAI 上板。缺点是对检测框宽度、目标姿态和相机焦距标定比较敏感，因此工程中需要通过多距离实测修正焦距参数，并对距离结果做滤波。

### 4.4 距离滤波

检测框每帧会有轻微抖动，直接使用框宽计算距离会导致距离显示和底盘速度跳动。本项目使用一阶低通滤波平滑距离值：

```text
filtered_distance = alpha * current_distance + (1 - alpha) * last_distance
```

该方法计算量小、参数少、调试直观。相比卡尔曼滤波，一阶低通滤波更适合当前阶段的实车快速调参。后续如果需要预测目标运动，可再升级为卡尔曼滤波。

### 4.5 图像中心瞄准控制

云台瞄准控制采用图像误差控制。设目标中心为 `(target_x, target_y)`，图像中心为 `(image_w / 2, image_h / 2)`，则归一化误差为：

```text
error_x = (target_x - image_w / 2) / (image_w / 2)
error_y = (target_y - image_h / 2) / (image_h / 2)
```

控制器根据误差生成 yaw、pitch 的位置增量：

```text
yaw_step = kp_yaw * error_x + kd_yaw * delta_error_x
pitch_step = kp_pitch * error_y + kd_pitch * delta_error_y
```

工程中没有直接使用完整 PID，而是使用 PD 思想，原因如下：

- 云台接口主要接收位置命令，不是直接控制电机电流或连续角速度。
- 检测框本身存在抖动，积分项容易累积误差并造成过冲。
- PD 控制能够在保持响应速度的同时抑制部分抖动。
- 输出最终会转换为整数位置增量，过复杂的浮点控制收益有限。

为保证实车安全，控制器还加入了死区和限幅：

- 目标接近图像中心时不再输出微小修正，减少抖动。
- 单帧最大位置增量受限，避免云台大幅跳动。
- yaw/pitch 方向可通过参数反向，便于适配实际安装方向。

### 4.6 固定距离跟随控制

小车底盘只能通过 CAN 接收整数转速命令，因此跟随控制采用距离误差到电机转速的比例映射：

```text
distance_error = current_distance - target_distance
rpm = kp_distance * distance_error
```

若目标距离大于设定距离，小车前进；若目标距离小于设定距离，小车后退；若距离误差处于死区内，则底盘停止或保持低速不动。

为了适配实车，本项目加入以下限制：

- 距离死区，避免目标接近设定距离时小车来回抖动。
- 最大速度限幅，避免小车突然高速运动。
- 最小有效速度，用于克服电机静摩擦。
- 左右电机方向符号可配置，用于适配接线和机械安装。
- 目标丢失后底盘速度清零。

## 5. 系统设计

### 5.1 模块结构

本项目核心代码位于 `aim_follow_control/`：

```text
aim_follow_control/
  include/aim_follow_controller.hpp
  src/aim_follow_controller.cpp
  test/aim_follow_controller_test.cpp
```

主要类和结构体如下：

| 名称 | 作用 |
| --- | --- |
| `TargetObservation` | 表示单帧检测目标，包括中心点、宽高、距离、置信度等 |
| `ControlConfig` | 保存控制参数，包括目标距离、增益、死区、限幅等 |
| `ControlOutput` | 保存输出命令，包括 yaw/pitch 增量、电机转速、状态标志 |
| `TargetSelector` | 负责从多个检测框中选择连续目标 |
| `AimFollowController` | 负责计算云台瞄准和底盘跟随命令 |

### 5.2 与原工程的数据接口

原工程 YOLO 后处理得到检测框后，应将检测框转换为 `TargetObservation`。建议使用 HDMI 显示映射后的检测框进行距离显示和框绘制，使用同一坐标系避免框位置错位。

推荐集成位置如下：

```text
YOLO 后处理
  -> 检测框坐标映射到显示坐标
  -> TargetSelector 选择当前目标
  -> 单目距离估计与滤波
  -> AimFollowController 计算控制输出
  -> 原工程 CAN 发送函数发送云台和底盘命令
```

这样可以保持原工程的摄像头、HDMI 和 CAN 主流程不变，只在检测结果和 CAN 输出之间插入算法模块。

### 5.3 控制输出映射

控制器本身只输出抽象命令：

- `yaw_step`
- `pitch_step`
- `left_rpm`
- `right_rpm`
- `has_target`
- `target_lost`

在原 PLin 工程中，这些命令需要映射到已有 CAN 协议：

- 云台控制帧继续使用原工程已有的云台 CAN ID 和数据格式。
- 底盘控制帧继续使用原工程已有的电机速度 CAN ID 和数据格式。
- CAN 波特率、设备名、初始化顺序保持原工程配置。

本项目不直接规定最终 CAN 字节序，而是通过集成说明要求复用原工程已经验证过的 CAN 发送函数，避免破坏原本可以工作的通信链路。

## 6. 参数整定方案

### 6.1 距离参数

距离估计需要实测以下参数：

- 目标实际宽度，例如自行车图片宽度 0.24 m。
- 目标在 0.5 m、1.0 m、1.5 m、2.0 m 等距离下的检测框像素宽度。
- 根据 `focal_length_px = distance_m * pixel_width / real_width_m` 计算焦距。
- 多组焦距取平均或剔除异常值后取稳定值。

实车中应优先保证检测框贴合目标真实宽度，否则距离误差会明显增大。

### 6.2 云台参数

云台参数建议按以下顺序调试：

1. 先设置较小 `yaw_kp` 和 `pitch_kp`，确认方向正确。
2. 若方向相反，调整 `invert_yaw` 或 `invert_pitch`。
3. 逐步增大比例增益，使目标能回到画面中心。
4. 加入少量微分增益，降低过冲。
5. 调整死区，减少目标接近中心后的抖动。
6. 调整最大步进，限制单帧大动作。

### 6.3 底盘参数

底盘跟随参数建议按以下顺序调试：

1. 先架空车轮，确认电机方向。
2. 设置较低最大转速，例如 100 至 160 rpm。
3. 设置目标距离，例如 1.0 m。
4. 逐步增大距离比例系数，使小车能缓慢靠近目标距离。
5. 调整距离死区，避免小车在目标距离附近反复前后动。
6. 实地测试 0.5 m、1.0 m、1.5 m、2.0 m 下的响应。

## 7. 上板验证流程

建议上板验证分四步完成：

1. 本地单元测试：验证目标选择、瞄准输出、距离跟随输出和丢失保护逻辑。
2. 部署前检查：确认模块已同步到 PLin 工程，CMake 已包含头文件目录和源文件。
3. 板端冒烟测试：运行主程序并保存日志，检查是否出现 `[AIM FOLLOW]`、`[DISTANCE DEBUG]` 等关键输出。
4. 实车安全测试：先架空车轮，再加入目标，确认方向和速度后再落地测试。

本项目提供以下脚本辅助验证：

```text
tools/run_local_aim_follow_checks.ps1
tools/sync_to_plin_project.ps1
tools/check_deploy_dry_run.ps1
tools/deploy_30tai.ps1
tools/analyze_smoke_logs.ps1
```

## 8. 参考算法与资料

本项目主要参考以下方向：

- YOLO 系列目标检测算法：用于实时输出目标框。
- 单目针孔相机测距模型：用于通过目标实际宽度和像素宽度估计距离。
- 视觉伺服控制思想：用于将图像中心误差转换为云台控制量。
- DeepSORT、ByteTrack 等多目标跟踪方法中的数据关联思想：用于设计轻量级目标连续选择策略。
- PID/PD 控制思想：用于目标中心误差到位置增量、距离误差到速度命令的映射。

考虑到 30TAI 的上板资源和当前工程结构，本项目没有直接移植完整 DeepSORT、ByteTrack 或复杂视觉伺服框架，而是保留其核心工程思想，形成轻量、可测、易集成的目标瞄准与定距跟随模块。
